# Model contract

Reference for the `Model` C++ struct that Exasim's templated FEM
internals consume. Every method, every signature, defaults, and
indexing layouts.

Tutorial walkthrough: [`tutorial/04-handwritten-cli/`](../tutorial/04-handwritten-cli/README.md).

Exasim's templated FEM internals (HDG residual assembly, GMRES solver,
preconditioner, time stepping, visualization, QoI integration) are
parameterized on a user-supplied `Model` type. After
`find_package(Exasim REQUIRED)`, the consumer pulls in the
`Exasim::headers` INTERFACE target and instantiates `CSolution<MyModel>`
from `<exasim/solution.hpp>`. No compiled Exasim library is installed
— the runtime exists only in your TU after the templated kernels are
inlined into your model.

The user supplies a `MyModel` struct that satisfies the contract in
`<exasim/model.hpp>`. The struct can be hand-written or generated by
`text2code` from a `pdemodel.txt` description.

## Minimum consumer CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.16)
project(my_solver CXX)
set(CMAKE_CXX_STANDARD 17)

find_package(Kokkos REQUIRED)
find_package(MPI    REQUIRED)
find_package(BLAS   REQUIRED)
find_package(LAPACK REQUIRED)
find_package(Exasim REQUIRED)

add_executable(my_solver main.cpp)
target_link_libraries(my_solver PRIVATE
    Exasim::headers Kokkos::kokkos MPI::MPI_CXX
    ${BLAS_LIBRARIES} ${LAPACK_LIBRARIES})
```

`Exasim::headers` deliberately does **not** propagate `find_dependency`
calls for Kokkos/MPI/BLAS/LAPACK. Downstream projects choose their own
Kokkos build directory, their own MPI implementation, and their own BLAS
— Exasim only requires that they be findable in your project.

## The Model contract

A user model is a plain C++ struct with a small set of compile-time
constants and `KOKKOS_INLINE_FUNCTION static` pointwise methods. Inherit
from `exasim::ModelDefaults<Self>` (CRTP) for zero-fill defaults of the
optional methods.

### Required compile-time constants

```cpp
struct MyModel : exasim::ModelDefaults<MyModel> {
    static constexpr int nd     = 2;       // spatial dimension
    static constexpr int ncu    = 1;       // # primary unknowns
    static constexpr int ncw    = 0;       // # auxiliary scalar fields (the `w` array)
    static constexpr int nco    = 0;       // # auxiliary "other DG" fields (the `v` array)
    static constexpr int nparam = 1;       // # physics parameters (the `mu` array)

    static constexpr auto disc  = exasim::Discretization::HDG;  // or LDG
};
```

### Required pointwise methods

The runtime aborts with a Newton non-convergence if any of these are
left as the default zero-fill — `MyModel` MUST override them:

| Method | Output | Purpose |
|---|---|---|
| `flux`     | `f[ncu*nd]` | volume flux at quadrature point |
| `source`   | `s[ncu]`    | volume forcing term |
| `fbou`     | `fb[ncu]`   | LDG boundary flux (per-`ib` dispatch via `if (ib == 1) ...`) |
| `fbou_hdg` | `fb[ncu]`   | HDG boundary residual (per-`ib` dispatch) |
| `ubou`     | `ub[ncu]`   | imposed boundary value (Dirichlet etc., per-`ib` dispatch) |
| `initu`    | `ui[ncu]`   | initial condition |

### HDG Jacobians (required for `disc = HDG`)

The HDG kernels need pointwise Jacobians. Defaults are zero-fill, which
makes the linearized system zero and Newton fail — for HDG you MUST
override:

| Method | Output | Layout |
|---|---|---|
| `flux_jac_uq`        | `f_uq[ncu*nd*Nq]`  | column-major: `f_uq[j*(ncu*nd) + i] = ∂f[i]/∂uq[j]` |
| `flux_jac_w`         | `f_w[ncu*nd*ncw]`  | column-major: `f_w[j*(ncu*nd) + i] = ∂f[i]/∂w[j]`   |
| `source_jac_uq`      | `s_uq[ncu*Nq]`     | column-major |
| `source_jac_w`       | `s_w[ncu*ncw]`     | column-major |
| `fbou_hdg_jac_uq`    | `fb_uq[ncu*Nq]`    | column-major (per-`ib` dispatch) |
| `fbou_hdg_jac_uh`    | `fb_uh[ncu*ncu]`   | column-major (per-`ib` dispatch) |
| `fbou_hdg_jac_w`     | `fb_w[ncu*ncw]`    | column-major (per-`ib` dispatch) |

`Nq = ncu * (1 + nd)` is the size of the `uq` array (primary unknowns
plus their gradients in HDG).

### Volume method signature

```cpp
KOKKOS_INLINE_FUNCTION static
void flux(double f[],
          const double x[],          // [nd]
          const double uq[],         // [Nq]
          const double v[],          // [nco]   (auxiliary odg field)
          const double w[],          // [ncw]
          const double mu[],         // [nparam] physics params
          const double uinf[],       // free-stream values (may be nullptr)
          double t)                  // current time
{
    f[0] = mu[0] * uq[1];
    f[1] = mu[0] * uq[2];
}
```

### Boundary method signature

```cpp
KOKKOS_INLINE_FUNCTION static
void fbou_hdg(double fb[], int ib,
              const double x[],   const double uq[],
              const double v[],   const double w[],   const double uh[],
              const double n[],   const double tau[],
              const double mu[],  const double uinf[], double t)
{
    if (ib == 1) {                 // first boundary tag
        fb[0] = -tau[0] * uh[0];
    }
    // else if (ib == 2) ...
}
```

### Optional methods (zero-fill defaults from ModelDefaults)

`sourcew`, `tdfunc`, `avfield`, `eos`, `eos_du`, `eos_dw`,
`fhat`, `uhat`, `stab`, `vis_scalars`, `vis_vectors`, `vis_tensors`,
`qoi_volume`, `qoi_boundary`, `monitor`, `output`, `initq`, `initwdg`,
`initudg`, `initodg`. Override only what your PDE actually defines.

## Minimum main.cpp

```cpp
#include <exasim/run.hpp>
#include "my_model.hpp"

int main(int argc, char** argv) {
    return exasim::run<MyModel>(argc, argv);
}
```

`exasim::run<M>(argc, argv)` is a templated façade that handles MPI /
Kokkos init+finalize, command-line parsing (`pdeapp.txt` path or the
older `nummodels InputFile OutputFile [restart]` form), DIRK time
stepping, multi-domain / multi-physics coupling, pseudo-time stepping,
solution I/O, and the QoI / Paraview save loop. It instantiates
`CSolution<M>` internally — your `MyModel` struct supplies the math.

The same façade powers `backend/Main/main.cpp` (legacy `cput2cEXASIM`
binaries, calling `exasim::run<exasim::detail::AbiAdapter>`), so the
templated and ABI paths share one body.

If you need to do MPI / Kokkos init yourself (e.g., interleaving Exasim
with your own pre/post-processing), include `<exasim/solution.hpp>`
directly and instantiate `CSolution<MyModel>` — see `<exasim/run.hpp>`'s
source for what it does between init and finalize.

## Two authoring paths

**Hand-written**: write `my_model.hpp` directly. See
`apps/library_example/poisson2d/my_model.hpp` for a worked example
(Poisson with the manufactured solution `sin(πx) sin(πy)`).

**text2code-generated**: write a `pdemodel.txt` describing the PDE as
SymEngine expressions, then run `text2code <pdeapp.txt>`. The same
header-only `my_model.hpp` is emitted — no `.so`, no link-time ABI.
See `apps/library_example/poisson2d_codegen/` for an example.

Both paths produce the same struct shape; the templated runtime
consumes them identically.

## What's installed

`find_package(Exasim REQUIRED)` exposes:

| Component | Header |
|---|---|
| FEM internals | `<exasim/{discretization,solver,preconditioner,solution,visualization}.hpp>` |
| Pointwise kernel templates | `<exasim/kernels/{flux,source,boundary,interface,init,tdfunc,eos,sourcew,visualization,qoi,output}.hpp>` |
| Driver wrappers | `<exasim/drivers.hpp>` |
| Model contract | `<exasim/model.hpp>` |
| CLI driver | `<exasim/run.hpp>` |
| Embedded API | `<exasim/solver_facade.hpp>` |

`Exasim::headers` is INTERFACE-only; no compiled library is shipped.
The runtime exists in your TU after the templated kernels inline
your pointwise math at build time.
